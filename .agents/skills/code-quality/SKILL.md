# Code Quality & Cohesion Skill

이 스킬은 프로젝트의 코드 품질, 특히 **높은 응집도(High Cohesion)**와 **낮은 결합도(Low Coupling)**를 유지하기 위한 설계 지침을 정의합니다.

## 1. 응집도(Cohesion) 향상 지침

### 1.1 Co-location (근접 배치)
- **원칙**: 함께 사용되는 것들은 최대한 가까이 둡니다.
- **실천**: 특정 컴포넌트에서만 사용하는 **타입, 상수, 작은 유틸 함수**는 별도 파일로 분리하지 않고 컴포넌트 파일 내부에 두거나, 같은 폴더 내에 배치합니다.
- **예시**: `LoginForm.tsx`에서만 쓰는 `LoginSchema`나 `LoginFormProps` 타입은 `LoginForm.tsx` 파일 최상단에 정의합니다.

### 1.2 SRP (Single Responsibility Principle)
- **원칙**: 하나의 함수, 컴포넌트, 모듈은 하나의 책임만 가집니다.
- **실천**: 컴포넌트가 너무 커지면(예: 300라인 이상) 비즈니스 로직은 **커스텀 훅**으로, 하위 UI는 **작은 컴포넌트**로 분리하여 응집도를 높입니다.

### 1.3 데이터와 로직의 결합
- **원칙**: 데이터 구조와 그 데이터를 처리하는 로직을 한데 모읍니다.
- **실천**: 특정 엔티티의 상태 변경 로직은 해당 엔티티의 `model/` 내부에 위치시켜, 외부에서 데이터 구조를 직접 조작하지 않고 정의된 로직을 통하게 합니다.

## 2. 결합도(Coupling) 최소화 지침

### 2.1 추상화 수준 통일
- 한 함수/컴포넌트 내에서는 비슷한 추상화 수준을 유지합니다. 저수준의 상세 구현(예: 직접 DOM 조작, 복잡한 정규식)은 유틸리티로 추상화하여 호출부의 가독성을 높입니다.

### 2.2 인터페이스 의존
- 모듈 간의 통신은 구현 상세가 아닌 **인터페이스(Props, Type)**를 통해 이루어지게 합니다.
- **FSD 규칙 준수**: 하위 레이어가 상위 레이어를 알지 못하게 하여 의존성 전염을 방지합니다.

### 2.3 제어 역전 (IoC) 활용
- 컴포넌트가 직접 데이터를 fetch하지 않고, 상위에서 주입(Props)받거나 `useQuery` 같은 선언적 훅을 사용하도록 하여 특정 데이터 소스와의 결합도를 낮춥니다.

## 3. 에이전트 자가 검토 체크리스트
- [ ] 이 함수/컴포넌트가 한 가지 일만 하고 있는가?
- [ ] 관련 있는 타입과 로직이 흩어져 있지 않고 근처에 모여 있는가?
- [ ] 외부 모듈에 너무 깊게 의존하고 있지는 않은가? (레이어 위반 확인)
- [ ] 코드의 추상화 수준이 일관된가?
